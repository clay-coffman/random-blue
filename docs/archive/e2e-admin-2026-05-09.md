## Bugs

### A1 ‚Äî Mobile: `/admin/users` clips the role dropdown off-screen
- **Severity:** major (mobile)
- **Repro:** At 375 px, sign in as superadmin or goeo_admin, open
  `/admin/users`, scroll to the table.
- **Expected:** Role pill, status pill, joined date, and the
  role-flip `<select>` should be reachable.
- **Actual:** The `<table>` keeps a fixed multi-column layout. Only
  Name + Email + a half-clipped role pill ("FOUNDE‚Ä¶", "GOEO_A‚Ä¶",
  "SUPERA‚Ä¶") fit. Status, Joined, and the role-flip Action column
  are off-screen with no horizontal scroll affordance. Admins on a
  phone cannot use the page's only mutation.
- **File hint:** `app/admin/users/page.tsx` table markup ‚Äî
  needs the same stacked-card collapse the claim queue list uses
  (`app/admin/submissions/page.tsx` already does this correctly at
  375). See `m04b-users-table.png`.

### A2 ‚Äî Mobile: `/admin/companies` and `/admin/resources` clip the Edit column
- **Severity:** major (mobile)
- **Repro:** At 375 px, open `/admin/companies` or `/admin/resources`.
- **Expected:** Reach the per-row Edit link.
- **Actual:** Companies table shows Company / Sector / Stage and
  cuts off Status + Action; Resources table shows Title and a
  truncated "entrepre/communiti‚Ä¶" Kind cell only. Tapping a row
  does navigate to edit (the row link still works), but a discoverable
  Edit affordance is invisible at this width.
- **File hints:** `app/admin/companies/page.tsx`, `app/admin/resources/page.tsx`
  / `app/admin/resources/_components/*`. See
  `m04b-companies-table.png` and `m04b-resources-table.png`.

### A3 ‚Äî `/companies/[slug]/edit` silently coerces seeded sector/stage to "‚Äî"
- **Severity:** major (data-corrupting)
- **Repro:** As superadmin, open any seeded company's edit page ‚Äî
  e.g. `/companies/adminify-ai/edit` after the QA approval flow, or
  any unclaimed company. Inspect Sector and Stage.
- **Expected:** The form pre-fills with the persisted value (e.g.
  Sector "B2B Software", Stage "bootstrapped" or "seed").
- **Actual:** Sector and Stage both render as `‚Äî` (the disabled
  default), even when the table on the previous screen reads
  "B2B Software / bootstrapped". The two `<select>` enums are
  `fintech | saas | ai | aerospace | bio | energy | consumer | security | manufacturing | other`
  and `idea | early | seed | growth | scale | mature` ‚Äî neither
  contains the seeded raw values, so the editor falls back to "‚Äî"
  and a "Publish changes" save would overwrite the row with empty
  sector/stage. (No save was performed during this QA pass to avoid
  corrupting data; the bug was reproduced by visual inspection of
  the loaded form.)
- **File hint:** `app/companies/[slug]/edit/*` form. Either widen the
  enum to include the seeded values (`B2B Software`, `bootstrapped`,
  etc.), or normalize the seed loader to the canonical enum, or
  preserve the existing value as a non-enumerated read-only display
  when it doesn't match. See `10-company-edit.png` (desktop).

### A4 ‚Äî Sign-out from `/admin` keeps the user on `/admin` (stale view)
- **Severity:** major (UX trust)
- **Repro:** As superadmin or goeo_admin, on `/admin`, open the
  account menu and click **Sign out**.
- **Expected:** Redirect to `/` (or `/sign-in`) so it's obvious the
  session ended.
- **Actual:** The Better Auth POST returns 200 and the cookie is
  cleared, but the page stays at `/admin` showing the previously
  rendered content. The next navigation re-runs the layout guard
  and correctly redirects to `/sign-in?next=/admin`, but until the
  user does anything they appear to still be in the admin console.
- **File hint:** the user-menu sign-out handler in
  `components/site-header/*` (or wherever the sidebar profile card
  triggers Better Auth `/api/auth/sign-out`). It should
  `router.replace("/")` after the call resolves, especially when
  the current pathname starts with `/admin`.

## UX polish (works, but rough)

### P1 ‚Äî `/admin/investors` has no "pending verification" sidebar pill
- **What:** The sidebar shows pending counts only for Claim queue
  and Intro requests (`pendingByRoute` in `app/admin/layout.tsx:54`).
  Investor verification has the same review shape, but a brand-new
  unverified investor is invisible to an admin until they
  proactively click into the section.
- **Suggestion:** Add an `unverified` count for investor profiles
  with `slug IS NOT NULL` (only the public-eligible ones).

### P2 ‚Äî Submission detail page: "Domain match" computes mismatch from email-vs-website hostname
- **What:** "Domain mismatch" is shown for `priya@persona.test` vs
  `https://Airbatch.io` ‚Äî true and useful ‚Äî but the UI gives no
  indication of *why* the domain is being compared (the user's
  email domain isn't required to match the company website in
  general). A one-line tooltip ("compared user's email domain against
  the company website hostname; useful as one signal among several")
  would help admins moderate without inferring intent.

### P3 ‚Äî Recent agent edits empty state on a fresh DB
- **What:** Dashboard "Recent agent edits" panel is empty by
  design when nothing has been edited yet. The "audit log ‚Üí"
  pseudo-link in the header is plain text, not a real link, even
  on hover. Either make it an anchor or drop it.
- **File:** `app/admin/page.tsx` (~ "Recent agent edits" header).

### P4 ‚Äî "Coverage gaps" widget is hardcoded placeholder data
- **What:** The four cards (`Washington Co. ‚Äî 2 ag resources ¬∑ low`,
  `Veteran tag ‚Äî 4 resources ¬∑ stale`, `Energy sector ‚Äî 7 cos ¬∑ 0
  hiring`, `Rural & women ‚Äî ‚Üë submission ready`) are literals in
  `app/admin/page.tsx`, not derived from the DB. The component already
  carries an honest "Phase 5 wires real coverage queries here." note
  underneath, so this is known ‚Äî but in production this section
  will lie to admins until the queries are wired. Consider hiding
  the section behind a feature flag or until at least one query is
  live.

### P5 ‚Äî Resource edit form drops most of the resource record
- **What:** `/admin/resources/[id]` exposes only ID, Title,
  Description, Source URL, Contact email, Kind. The seeded
  resource rows carry stage/persona/county/tag affinities that
  drive the recommendation engine but are completely uneditable
  here. Today an admin can fix a typo but can't re-tune which
  founders see a resource ‚Äî that's a content-ops gap. Even a
  read-only display of the affinities (with a "edit via SQL / agent
  CLI" pointer) would help.

## Dev-DX

### D1 ‚Äî R2 document preview unavailable in dev
- **Severity:** dev-DX (admin reviews the document blind)
- **Repro:** `/admin/submissions/[id]`. The Document preview block
  reads:
  > Preview unavailable: R2 S3 credentials missing. Set
  > `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`.
- **Why it matters:** The whole point of the manual review is to
  inspect the uploaded LLC articles / EIN letter. With no preview
  in dev an admin clicks Approve based purely on the metadata
  (claimant email, domain match, file size, mime). The flow is
  *technically* exercisable but it's a degraded surface for any
  agent QA-ing the page.
- **Suggestions:** either auto-fall-back to a wrangler R2 local
  binding (the preview URL helper could detect missing creds and
  emit a "[dev ‚Äî view file at `.wrangler/.../<key>`]" link), or
  leave a louder banner pointing at the env-var instructions in
  CLAUDE.md.
- **File hint:** wherever the signed-URL helper is invoked from
  `app/admin/submissions/[id]/page.tsx`.

### D2 ‚Äî Sign-up password rule surprises new admins
- **Severity:** dev-DX (also user-facing for *every* new sign-up,
  but the only screen admins go through is during invite acceptance)
- **Repro:** Accept the admin invite as `qa-invite@example.test`.
  At the account form, type `passport123` (the seed test password).
- **Expected:** Submit, given `AUTH_SKIP_OTP=true`.
- **Actual:** The "weak / good" indicator marks the password "weak"
  and the form refuses to advance with a "Use at least 12
  characters" hint right next to the field. This is fine for
  production ‚Äî but the seed users in `db/seed/users.ts` use the same
  `passport123` (11 chars) and *they* sign in cleanly because their
  hash was built directly. New admins accepting an invite have to
  invent a longer password.
- **Why it matters:** anyone copying the test-credential pattern
  from the seed file when accepting a real invite will hit a
  validation wall. Either drop the seed example up to 12+ chars
  ("passport12345") or note the rule explicitly in CLAUDE.md ¬ß
  Local authentication testing alongside the
  `AUTH_SKIP_OTP=true` pointer.

## Cross-references to the prior founder doc

`docs/e2e-findings-2026-05-09.md` already filed **B4** (sign-up still
routes to `/sign-up/verify` even with `AUTH_SKIP_OTP=true`). That bug
is also load-bearing for the admin-invite flow ‚Äî accepting an admin
invite goes through the same sign-up funnel, so an invitee who
follows the email link, hits "Create account", and tries to land on
`/invite/<token>` is detoured through `/sign-up/verify` first. In
this QA pass we worked around it by manually navigating back to the
invite URL after submitting the account form (the user is already
authenticated by then). That detour should be fixed by the same
patch that closes B4.
